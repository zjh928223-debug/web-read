## ADDED Requirements

### Requirement: 在模块拆分前必须完成运行时结构盘点
系统在对阅读器进行模块拆分或重构前，项目维护流程 MUST 先基于真实仓库代码完成运行入口、渲染逻辑、状态管理、存储/持久化、播放器/转录/笔记/切块/交互事件和共享辅助函数的结构盘点，并形成可复用的分析结论。

#### Scenario: 识别主运行文件
- **WHEN** 团队准备开始新的拆分或重构 change
- **THEN** 必须明确当前实际运行入口文件、核心运行文件和辅助模块文件
- **THEN** 不得只基于 `openspec/` 或历史文档推断结构

#### Scenario: 识别职责混杂区域
- **WHEN** 团队盘点当前结构
- **THEN** 必须标出承担过多职责的文件或模块
- **THEN** 必须记录明显耦合严重、功能缠绕或半删除残留的区域

### Requirement: safe split 路线必须按风险从低到高排序
系统的模块拆分计划 SHALL 以保持现有行为稳定为首要目标，并按低风险到高风险给出分阶段顺序。前置阶段必须优先考虑纯工具函数、纯数据校验、纯存储辅助和共享小函数，避免在缺少边界时直接进入核心渲染和交互逻辑。

#### Scenario: 规划第一阶段拆分
- **WHEN** 团队定义第一批拆分目标
- **THEN** 应优先选择可参数化、低副作用、与 UI 行为耦合较弱的辅助层
- **THEN** 不应优先拆 `renderChunkMode()`、`mainUpdateHighlight()` 或等价的核心行为心跳

#### Scenario: 标记高风险区域
- **WHEN** 团队输出拆分顺序
- **THEN** 必须明确哪些区域属于高风险拆分对象
- **THEN** 必须说明其高风险原因，例如共享全局状态、播放时序、渲染与交互混合或历史残留耦合

### Requirement: 结构规划变更不得隐式改变现有运行行为
以结构认知和拆分规划为目标的 change MUST 保持现有阅读器行为不变，不得在未单独建 change 的情况下顺带重写主题、播放器、高亮、chunk note、句注或其他业务逻辑。

#### Scenario: 规划 change 输出
- **WHEN** 团队创建与结构盘点相关的 OpenSpec change
- **THEN** 该 change 应聚焦于认知、边界和任务规划
- **THEN** 不应把业务修复、UI 重写或核心交互重构混入同一 change
